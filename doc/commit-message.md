# コミットメッセージ規約

このリポジトリでは、コミットメッセージを以下の形式で記述します。

## 基本形式

```text
<type>(<scope>): <subject>

<body>

<footer>
```

- `<type>`: 変更の種類（必須）
- `(<scope>)`: 影響範囲（任意）
- `<subject>`: 1行要約（必須）
- `<body>`: 背景・意図・実装方針（任意）
- `<footer>`: Issue参照や破壊的変更の注記（任意）

## type 一覧

- `feat`: ユーザー向け機能追加・仕様追加
- `fix`: バグ修正
- `refactor`: 振る舞いを変えない内部改善
- `perf`: 性能改善
- `docs`: ドキュメントのみ変更
- `test`: テスト追加・修正
- `build`: ビルド設定や依存関係変更
- `ci`: CI/CD 設定変更
- `chore`: 雑多な保守作業（動作影響が小さいもの）
- `revert`: 既存コミットの取り消し

## subject のルール

- 50文字以内を目安に簡潔に書く
- 文末の句点（。や.）は付けない
- 何を変えたかが一目で分かる表現にする
- 日本語で記述する（このリポジトリの運用に合わせる）

良い例:

```text
fix(layout): 縦書き時にカーソルが右へずれる不具合を修正
```

悪い例:

```text
fix: いろいろ修正
```

## body のルール（必要な場合のみ）

- 「なぜ必要か」と「どう直したか」を中心に書く
- 1行は72文字程度までを目安に折り返す
- 実装手順の羅列より、判断理由を優先する

例:

```text
入力中の IME preedit がレイアウト計算に含まれず、
確定前文字列の表示位置が1文字分ずれていた。

preedit 用の幅計算を通常文字列と同じ経路に統一し、
表示位置とカーソル位置の差分を解消した。
```

## footer のルール

- Issue 連携は `Refs #123` または `Closes #123` を使う
- 破壊的変更は `BREAKING CHANGE: <内容>` を使う

例:

```text
Closes #245
```

## scope の使い方（任意）

- crate名や機能名を短く入れる
- 迷ったら省略してよい

例:

- `feat(kashikishi): ...`
- `fix(font_rasterizer): ...`
- `docs: ...`

## コミット粒度

- 1コミット1目的を徹底する
- リファクタと機能追加は可能な限り分ける
- フォーマットのみ変更する場合は機能変更と分ける

## 例

```text
feat(kashikishi): メモ一覧にタグ絞り込みを追加

fix(text_buffer): 改行直後にUndoするとカーソル位置が壊れる不具合を修正

refactor(ui_support): 入力イベント変換ロジックをモジュール分割

docs: feature-dev ワークフローにチェック手順を追記
```
